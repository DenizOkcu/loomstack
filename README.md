# loom

**loom is an agent-operable fullstack framework.** Humans specify product logic; coding agents work inside an explicit, generated, verifiable architecture.

> Less ambiguity. Fewer wrong edits. Predictable fullstack changes.

loom v0.1 has one golden path: TypeScript, pnpm, React + Vite, Koa, SQLite, Zod-backed schemas, and Vitest. It deliberately does not offer alternate adapters or multiple valid application structures.

## The contract

```text
React        = presentation
Koa          = transport
features/*   = product behavior and source of truth
generated    = wiring; never hand-edit
```

Every feature declares its entities, routes, actions, queries, views, permissions, and tests. loom generates registries and transport, exposes scoped context to agents, and rejects architecture drift with stable repairable errors.

## Repository quickstart

Requirements: Node 22+ and pnpm 11+.

```bash
pnpm install
pnpm test
pnpm build

# Create and inspect an app from this checkout
pnpm loom create app demo-app
pnpm loom --cwd demo-app create feature people
pnpm loom --cwd demo-app context feature people --json
pnpm loom --cwd demo-app generate
pnpm loom --cwd demo-app verify --json
```

Published packages will let generated apps use their own scripts directly:

```bash
pnpm loom create feature people
pnpm loom generate
pnpm loom verify --json
```

## Agent workflow

For any change, an agent follows one deterministic loop:

```bash
pnpm loom context feature <feature> --json
pnpm loom affected <authored-file> --json
# edit feature.yaml/schema/action/query/view/test files
pnpm loom generate
pnpm loom verify feature <feature> --json
pnpm test
```

Generated source begins with:

```ts
// GENERATED BY loom. DO NOT EDIT.
```

Generated JSON has `"generatedBy": "loom"` and `"doNotEdit": true`. Hashes in `.loom/generated-files.json` distinguish manual edits from stale output.

## Canonical feature

```text
features/<feature-id>/
  feature.yaml
  AGENTS.md
  model.schema.ts
  permissions.policy.ts
  actions/*.action.ts
  queries/*.query.ts
  ui/*.view.tsx
  ui/*.component.tsx
  ui/*.form.tsx
  tests/*.test.ts
```

Mutations use `action()`. Reads use `query()`. Runtime input, output, and authentication checks are centralized. React views call named generated RPC clients. Koa exposes only:

```text
POST /_loom/actions/<actionName>
POST /_loom/queries/<queryName>
```

## CLI

All agent-facing commands support `--json`.

| Command | Purpose |
|---|---|
| `loom create app <name>` | Create the complete golden-path application |
| `loom create feature <name>` | Scaffold one canonical feature and refresh generation |
| `loom generate` | Regenerate routes, clients, registries, graph, context, and hashes |
| `loom verify [feature <name>]` | Enforce manifests, boundaries, and generated freshness |
| `loom context [feature <name>]` | Return scoped edit context |
| `loom graph` | Return the feature graph |
| `loom affected <file>` | Return likely related authored files |
| `loom explain <code>` | Return the reason and repair for a stable error |
| `loom doctor` | Check Node, pnpm, config, and TypeScript setup |

See [docs/commands.md](docs/commands.md) for output contracts.

## Packages

- `@loom/core` — manifests, project scanning, graph, errors
- `@loom/runtime` — schemas, actions, queries, policies, execution
- `@loom/generator` — deterministic app, feature, route, registry, and context generation
- `@loom/verifier` — architecture and freshness enforcement
- `@loom/react` — views, hooks, generated RPC client primitives
- `@loom/koa` — action/query HTTP execution
- `@loom/sqlite` — Node SQLite and deterministic in-memory test adapter
- `@loom/cli` — command surface for agents
- `@loom/create-app` — initializer binary

## Proof app

[`examples/manager-crm`](examples/manager-crm) contains people, projects, and commitments vertical slices. Commitments include due dates, overdue-first ordering, and a due-soon query.

```bash
pnpm loom --cwd examples/manager-crm verify --json
pnpm --filter manager-crm test
pnpm --filter '@manager-crm/*' build
```

## Scope

v0.1 intentionally excludes SSR, production auth/migrations, REST generation, plugins, MCP, deployment adapters, alternate UI/backend stacks, and advanced relation modeling. Those do not block proving the agent-operability thesis.

The original product and implementation specification remains in [`implementation-plan/`](implementation-plan/README.md).

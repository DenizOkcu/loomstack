# loomstack framework agent instructions

loomstack is an agent-operable TypeScript fullstack framework. Product behavior belongs in feature slices; React is presentation; Koa is transport.

## Golden path

- TypeScript and pnpm workspaces
- React + Vite
- Koa
- PostgreSQL
- Zod-backed schemas
- Vitest

## Hard rules

- Prefer explicit contracts and one canonical pattern.
- Generate transport and route wiring; never place behavior in generated files.
- Every generated source file starts with `// GENERATED BY loomstack. DO NOT EDIT.`.
- Every agent-facing command supports `--json`.
- Errors have stable codes and repair guidance.
- Generated output is deterministic.
- Do not add alternative adapters, plugins, SSR, REST generation, or production auth in v0.1.

## Commands

```bash
pnpm test
pnpm typecheck
pnpm build
pnpm loomstack --help
pnpm loomstack --cwd examples/manager-crm verify --json
```

## Efficient agent operation

Use LoomStack's structured commands before broad repository exploration. All agent-facing commands should use `--json`.

1. For framework changes, identify the owning package below and inspect its focused tests.
2. For generated-app behavior, query the proof app first:
   ```bash
   pnpm loomstack --cwd examples/manager-crm context --json
   pnpm loomstack --cwd examples/manager-crm graph --json
   ```
3. For feature-specific work, narrow context and impact before editing:
   ```bash
   pnpm loomstack --cwd examples/manager-crm context feature <name> --json
   pnpm loomstack --cwd examples/manager-crm affected <authored-file> --json
   ```
4. Regenerate only through the CLI, then verify scoped before full:
   ```bash
   pnpm loomstack --cwd examples/manager-crm generate --json
   pnpm loomstack --cwd examples/manager-crm verify feature <name> --json
   ```
5. If a command returns a stable code, inspect its contract instead of bypassing it:
   ```bash
   pnpm loomstack explain <code> --json
   ```
6. Finish with the smallest relevant tests, then `pnpm test`, `pnpm typecheck`, and `pnpm build` for framework-wide changes.

Use `context` instead of manually collecting broad context, `graph` for cross-feature or route questions, `affected` to find companion files, `create feature` rather than hand-built feature folders, `generate` for all derived wiring, and `doctor` for environment/setup failures. Never hand-edit generated fixtures; change generator sources and regenerate them.

## Package ownership

- `core`: manifests, scanning, graphs, errors, project configuration
- `runtime`: schemas, actions, queries, policies, execution
- `generator`: app/feature scaffolding and all derived files
- `verifier`: architecture and generated-file enforcement
- `react-adapter`: views, hooks, RPC client
- `koa-adapter`: RPC transport
- `postgres-adapter`: v0.1 persistence
- `cli`: agent-facing command surface

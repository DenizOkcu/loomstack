# 04 — Schema and Domain Layer

The schema layer defines entities, validation, input/output contracts, and generated types.

loomstack should be schema-first because schemas reduce ambiguity for agents.

## Goals

- One canonical way to define entities.
- One canonical way to define action/query input and output.
- Types are generated or inferred from schemas.
- Validation is automatic at runtime boundaries.
- Schema changes produce predictable affected files.

## v0.1 recommendation

Use a thin schema API backed by Zod internally or use Zod directly behind framework helpers.

The public API should look loomstack-native even if the implementation uses Zod.

Example:

```ts
import { entity, id, text, boolean, timestamp, userId } from "@loomstack/runtime"

export const PersonSchema = entity("Person", {
  id: id(),
  name: text().min(1).max(120),
  role: text().optional(),
  ownerId: userId(),
  createdAt: timestamp(),
  updatedAt: timestamp()
})
```

## Entity API

Required helpers for v0.1:

```ts
entity(name, fields)
schema(fields)
id()
text()
number()
boolean()
timestamp()
userId()
array(item)
oneOf(values)
optional(field)
```

It is acceptable for these to wrap Zod.

## Generated/inferred types

Users should be able to write:

```ts
export type Person = InferEntity<typeof PersonSchema>
```

or the framework can generate:

```ts
export type Person = {
  id: string
  name: string
  role?: string
  ownerId: string
  createdAt: Date
  updatedAt: Date
}
```

For v0.1, prefer type inference over file generation unless generated types are needed for app ergonomics.

## Input schema example

```ts
import { schema, text } from "@loomstack/runtime"

export const CreatePersonInput = schema({
  name: text().min(1).max(120),
  role: text().optional()
})
```

## Output schema example

Outputs can reference entity schemas:

```ts
export const PersonOutput = PersonSchema
```

List outputs can use arrays:

```ts
export const PeopleListOutput = schema({
  people: array(PersonSchema)
})
```

## Schema registry

The generator should create a schema registry:

```ts
// GENERATED BY loomstack. DO NOT EDIT.

import { PersonSchema } from "../../features/people/model.schema"

export const schemaRegistry = {
  Person: PersonSchema
}
```

## Database mapping

For v0.1, keep database mapping simple.

Option A: generate PostgreSQL table definitions from entity schemas.

Option B: require minimal explicit database adapter definitions.

Preferred v0.1 compromise:

- entity schemas define app-level shape
- PostgreSQL adapter can generate simple tables for supported field types
- advanced migrations are non-goal

## Migration expectations

v0.1 does not need production-grade migrations.

Minimum acceptable behavior:

- `loomstack db generate` creates a PostgreSQL schema file
- `loomstack db apply` applies it for local development
- verifier can detect when entity schemas changed but generated DB schema is stale

## Agent impact of schema changes

When a schema changes, `loomstack affected <file>` should report likely related files.

Example:

```json
{
  "changedFile": "features/people/model.schema.ts",
  "affected": [
    "features/people/actions/create-person.action.ts",
    "features/people/actions/update-person.action.ts",
    "features/people/queries/list-people.query.ts",
    "features/people/ui/person-form.view.tsx",
    "features/people/tests/create-person.test.ts"
  ]
}
```

## Verifier rules

The verifier should catch:

- invalid entity names
- missing schema exports declared in manifest
- action input/output without schema
- query input/output without schema where applicable
- unsupported field types for PostgreSQL generation
- stale generated schema registry

## Non-goals for v0.1

- polymorphic entity inheritance
- complex relation modeling
- multi-database abstraction
- advanced migrations
- schema decorators
- dynamic runtime schema loading

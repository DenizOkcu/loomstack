# 05 — Action/Query Runtime

loom uses an action/query model for product behavior.

Actions mutate state. Queries read state.

This model is more agent-operable than arbitrary controllers, services, hooks, server actions, and routes because it gives agents one canonical place for behavior.

## Core rule

```txt
Mutations go in actions.
Reads go in queries.
Transport is generated.
```

## Action API

Example:

```ts
import { action, schema, text } from "@loom/runtime"
import { PersonSchema } from "../model.schema"

const CreatePersonInput = schema({
  name: text().min(1).max(120),
  role: text().optional()
})

export const createPerson = action({
  name: "createPerson",
  input: CreatePersonInput,
  output: PersonSchema,
  auth: "authenticated",

  async run(ctx, input) {
    return ctx.db.person.create({
      data: {
        name: input.name,
        role: input.role,
        ownerId: ctx.user.id
      }
    })
  }
})
```

## Query API

Example:

```ts
import { query, schema, array } from "@loom/runtime"
import { PersonSchema } from "../model.schema"

const ListPeopleOutput = schema({
  people: array(PersonSchema)
})

export const listPeople = query({
  name: "listPeople",
  output: ListPeopleOutput,
  auth: "authenticated",

  async run(ctx) {
    const people = await ctx.db.person.findMany({
      where: { ownerId: ctx.user.id },
      orderBy: { createdAt: "desc" }
    })

    return { people }
  }
})
```

## Runtime types

Suggested core types:

```ts
export interface ActionDefinition<TInput, TOutput> {
  kind: "action"
  name: string
  input: Schema<TInput>
  output: Schema<TOutput>
  auth?: AuthRequirement
  run(ctx: ActionContext, input: TInput): Promise<TOutput>
}

export interface QueryDefinition<TInput, TOutput> {
  kind: "query"
  name: string
  input?: Schema<TInput>
  output: Schema<TOutput>
  auth?: AuthRequirement
  run(ctx: QueryContext, input: TInput): Promise<TOutput>
}
```

## Runtime execution

Execution should be centralized.

```ts
export async function executeAction(def, ctx, rawInput) {
  const input = def.input.parse(rawInput)
  await assertAuth(ctx, def.auth)
  const result = await def.run(ctx, input)
  return def.output.parse(result)
}
```

```ts
export async function executeQuery(def, ctx, rawInput) {
  const input = def.input ? def.input.parse(rawInput) : undefined
  await assertAuth(ctx, def.auth)
  const result = await def.run(ctx, input)
  return def.output.parse(result)
}
```

## Context object

The runtime context should include:

```ts
export interface loomRequestContext {
  requestId: string
  user?: loomUser
  db: loomDatabase
  logger: loomLogger
}
```

## Auth requirements

v0.1 can support simple string auth modes:

```ts
type AuthRequirement = "public" | "authenticated"
```

Feature policies can later add ownership checks.

## Policy integration

v0.1 policy example:

```ts
export const peoplePolicy = policy({
  canRead(ctx, person) {
    return ctx.user?.id === person.ownerId
  },

  canWrite(ctx, person) {
    return ctx.user?.id === person.ownerId
  }
})
```

Actions and queries can call policies explicitly at first.

Later, loom can make policy wiring more declarative.

## Transport independence

Actions and queries should not depend on Koa.

Bad:

```ts
async run(koaCtx, input) {}
```

Good:

```ts
async run(ctx, input) {}
```

Koa-specific request data should be converted into `loomRequestContext` by the Koa adapter.

export const actionRegistry = {
  createPerson,
  updatePerson

```

export const queryRegistry = {

  getPerson

```

## Verifier rules

The verifier should catch:

- action declared in manifest but not exported
- exported action missing from manifest
- query declared in manifest but not exported
- exported query missing from manifest
- duplicate action/query names
- action without input schema
- action/query without output schema
- action/query names that do not match registry names
- Koa imports inside action/query files

## Non-goals for v0.1

- background jobs
- queues
- subscriptions
- streaming
- transactions as first-class DSL
- advanced auth provider integration

# 06 — React Frontend Adapter

The React adapter provides the presentation layer for loomstack.

React is the golden path frontend for v0.1, but React should remain constrained. loomstack is not a generic React architecture playground.

## Core rule

```txt
React files render UI and call typed generated clients. They do not own product behavior.
```

## Allowed UI file types

```txt
*.view.tsx       route-level screen
*.component.tsx  reusable feature-local component
*.form.tsx       form component
```

## View API

Example:

```tsx
import { view } from "@loomstack/react"
import { listPeople } from "../queries/list-people.query"

export default view({
  name: "PeopleListView",
  query: listPeople,

  render({ data }) {
    return (
      <Page title="People">
        <DataList
          items={data.people}
          empty="No people yet"
          renderItem={(person) => <PersonRow person={person} />}
        />
      </Page>
    )
  }
})
```

## Agent-facing view constraints

- Views must use `view({ ... })`.
- Route-level views must be declared in `feature.yaml`.
- Views should declare query dependencies.
- Views should use generated action/query clients or framework wrappers.
- Views must not import database modules.
- Views must not import Koa modules.
- Views must not call raw `fetch`.
- Views must not define global routes manually.

## Generated React routes

Generated route file:

```tsx
// GENERATED BY loomstack. DO NOT EDIT.

import { lazy } from "react"

export const routes = [
  {
    path: "/people",
    component: lazy(() => import("../../../features/people/ui/people-list.view"))
  },
  {
    path: "/people/:id",
    component: lazy(() => import("../../../features/people/ui/person-detail.view"))
  }
]
```

## Route generation rules

Routes come from all feature manifests.

The generator should:

1. scan `features/*/feature.yaml`
2. validate route declarations
3. resolve view exports
4. sort routes deterministically
5. write `apps/web/src/routes.generated.tsx`

## Typed clients

Generated client API should expose named functions for queries/actions.

Example:

```ts
// GENERATED BY loomstack. DO NOT EDIT.

export const api = {
  actions: {
    createPerson: createActionClient("createPerson")
  },
  queries: {
    listPeople: createQueryClient("listPeople")
  }
}
```

## Hooks

The React adapter can expose hooks:

```ts
useloomstackQuery(listPeople, input)
useloomstackAction(createPerson)
```

For v0.1, keep hooks simple and avoid introducing a complex cache system. It is acceptable to integrate TanStack Query later, but not necessary for the first proof.

## Forms

loomstack should encourage schema-backed forms but does not need a full form generator in v0.1.

Minimum pattern:

```tsx
export function PersonForm({ onSubmit }) {
  return (
    <Form schema={CreatePersonInput} onSubmit={onSubmit}>
      <TextField name="name" label="Name" />
      <TextField name="role" label="Role" />
      <SubmitButton>Create person</SubmitButton>
    </Form>
  )
}
```

## Generated app shell

The template should include:

```txt
apps/web/src/main.tsx
apps/web/src/app.tsx
apps/web/src/routes.generated.tsx
apps/web/src/api-client.generated.ts
```

The app shell should be minimal and stable.

## Verifier rules

The verifier should detect:

- raw `fetch(` in `features/*/ui/**/*`
- database imports in UI files
- Koa imports in UI files
- route view declared in manifest but missing file/export
- view file not declared in manifest when it is route-level
- manual edits to generated route/client files
- stale generated route/client files

## Non-goals for v0.1

- server-side rendering
- Next.js-style routing
- complex suspense/data cache architecture
- theming system
- component library
- automatic form layout generation
- visual editor

# 01 — Architecture Overview

## High-level architecture

loom apps separate product behavior from frontend and backend adapters.

```txt
features/*       = product behavior and source of truth
React adapter    = presentation runtime
Koa adapter      = HTTP transport runtime
SQLite adapter   = persistence runtime
CLI              = project generation, inspection, verification
Verifier         = architecture rule enforcement
Context system   = agent-readable project summaries
```

## Core zones

```txt
React = presentation
Koa = transport/runtime
Feature files = product logic
```

Agents should normally edit feature files, not adapter internals.

## Monorepo architecture

loom itself should be implemented as a pnpm workspace.

```txt
packages/
  cli/
  core/
  runtime/
  verifier/
  generator/
  react-adapter/
  koa-adapter/
  sqlite-adapter/
  create-loom-app/
examples/
  manager-crm/
templates/
  default-app/
```

## Package responsibilities

### `packages/core`

Owns framework data structures and shared types:

- FeatureManifest
- FeatureGraph
- loomProject
- FrameworkError
- ContextDocument
- command result schemas

No React, Koa, or database-specific code belongs here.

### `packages/runtime`

Owns the runtime primitives used in app code:

- `entity()`
- `schema()`
- `action()`
- `query()`
- `policy()`
- `view()`
- runtime context interfaces

This package is imported by generated apps.

### `packages/cli`

Owns the `loom` executable:

- command parsing
- project discovery
- create app
- create feature
- context
- graph
- affected
- verify
- explain

The CLI should delegate core logic to packages rather than owning it directly.

### `packages/generator`

Owns deterministic file generation:

- app template generation
- feature scaffolding
- React route generation
- Koa route generation
- context file generation
- AGENTS.md / CLAUDE.md generation

### `packages/verifier`

Owns architecture validation:

- manifest validation
- file naming validation
- dependency boundary validation
- generated file freshness checks
- forbidden import checks
- route consistency checks
- schema/action/query consistency checks where feasible

### `packages/react-adapter`

Owns React-specific runtime helpers:

- generated route integration
- typed query hooks
- typed action hooks
- view wrapper
- client API transport

### `packages/koa-adapter`

Owns Koa-specific runtime helpers:

- route registration
- request context creation
- body parsing integration
- action/query execution over HTTP
- structured error responses

### `packages/sqlite-adapter`

Owns SQLite persistence integration for v0.1.

Keep this simple. Avoid building a full ORM in v0.1 unless necessary.

### `packages/create-loom-app`

Optional npm initializer package.

Should eventually support:

```bash
pnpm create loom-app my-app
```

For v0.1, this can wrap the CLI create command.

## App source-of-truth model

Feature files are source of truth.

Generated files are derived from:

- `features/*/feature.yaml`
- schema declarations
- action/query exports
- route definitions
- project config

Generated files must contain a marker:

```ts
// GENERATED BY loom. DO NOT EDIT.
```

The verifier should fail if generated files are stale.

## Data flow

### Read flow

```txt
React view
  -> generated query client
  -> HTTP POST /_loom/queries/<queryName>
  -> Koa generated route
  -> query runtime
  -> database adapter
  -> output schema validation
  -> response
```

### Mutation flow

```txt
React view/form
  -> generated action client
  -> HTTP POST /_loom/actions/<actionName>
  -> Koa generated route
  -> input schema validation
  -> auth/policy check
  -> action runtime
  -> database adapter
  -> output schema validation
  -> response
```

## Agent-operable invariant

Any app behavior should be traceable through named contracts:

```txt
feature.yaml -> schema -> action/query -> generated route -> generated client -> view
```

When this trace breaks, `loom verify` should report it.
